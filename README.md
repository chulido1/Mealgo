# Mealgo

Mealgo — desktop-first веб-приложение агрегатора доставки еды из разных ресторанов, разработанное как дипломный проект на PHP 8, MySQL и чистом JavaScript.

Проект моделирует работу коммерческого сервиса доставки: пользователь выбирает ресторан, открывает меню, добавляет блюда в отдельную корзину конкретного ресторана, оформляет заказ, управляет профилем, адресами, избранным и историей заказов. Администратор управляет ресторанами, меню, пользователями, заказами, отзывами и логами.

## Роли

В системе используются три роли:

- `guest` — гость, который может просматривать рестораны и меню, регистрироваться и входить.
- `user` — авторизованный пользователь, который может оформлять заказы, добавлять избранное, управлять профилем и адресами.
- `admin` — администратор, который управляет содержимым сайта через админ-панель.

## Установка через XAMPP

1. Скопируйте проект в папку:

```text
C:\xampp\htdocs\diplomv2
```

2. Запустите в XAMPP:

- Apache
- MySQL

3. Откройте phpMyAdmin:

```text
http://localhost/phpmyadmin
```

4. Создайте базу данных:

```sql
CREATE DATABASE diplom_foodhub CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

5. Импортируйте актуальный дамп:

```text
database/mealgo_full_dump.sql
```

6. Проверьте параметры подключения в файле:

```text
config/database.php
```

По умолчанию используются настройки XAMPP:

```text
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=diplom_foodhub
DB_USERNAME=root
DB_PASSWORD=
```

7. Откройте сайт:

```text
http://localhost/diplomv2/public
```

## Корзины

В проекте реализована логика нескольких активных корзин:

- один пользователь может иметь несколько активных корзин;
- каждая корзина привязана только к одному ресторану;
- если пользователь добавляет блюдо из нового ресторана, создается отдельная корзина;
- оформление заказа выполняется отдельно для каждой корзины;
- в шапке отображается количество товаров в корзинах;
- доступна выезжающая панель корзин.

Ключевые таблицы:

- `carts`
- `cart_items`
- `orders`
- `order_items`

## Email и MailHog

Email используется для:

- подтверждения регистрации кодом;
- повторной отправки кода подтверждения;
- восстановления пароля;
- уведомления о подписке на новости.

Для локальной разработки используется MailHog:

```text
MAIL_DRIVER=smtp
MAIL_HOST=127.0.0.1
MAIL_PORT=1025
```

Интерфейс MailHog обычно доступен по адресу:

```text
http://localhost:8025
```

Если MailHog не запущен, письма не будут отображаться во входящих, но логика генерации кодов и токенов остается рабочей.

## Подтверждение email

После регистрации:

1. Создается пользователь.
2. Генерируется код подтверждения.
3. Код сохраняется в таблицу `email_verification_codes`.
4. Код отправляется на email.
5. Пользователь вводит код на странице подтверждения.
6. При корректном коде поле `email_verified_at` заполняется текущей датой.

Если пользователь не подтвердил email, на сайте отображается уведомление с переходом на страницу подтверждения.

## Восстановление пароля

Восстановление пароля работает через токен:

1. Пользователь вводит email.
2. Создается запись в таблице `password_reset_tokens`.
3. На email отправляется письмо со ссылкой.
4. Пользователь открывает ссылку и задает новый пароль.
5. Токен помечается использованным.

Ссылка восстановления действует ограниченное время.

## Требования к паролю

Пароль должен содержать:

- минимум 8 символов;
- минимум одну заглавную букву;
- минимум один специальный символ.

Проверка выполняется на backend-стороне.

Если внешний API недоступен, приложение может вернуть локальные подсказки, чтобы форма адреса не ломалась во время демонстрации.

## Защита от SQL-инъекций

Все запросы к базе данных выполняются через PDO.

Для пользовательского ввода используются подготовленные выражения:

```php
$stmt = db()->prepare('SELECT * FROM users WHERE email = :email');
$stmt->execute(['email' => $email]);
```

Это предотвращает выполнение пользовательского ввода как SQL-кода.

## Защита от XSS

Пользовательские данные выводятся через функцию `e()`:

```php
<?= e($user['name']) ?>
```

Функция использует `htmlspecialchars`, поэтому HTML/JS-код из пользовательского ввода не выполняется в браузере.

## CSRF-защита

Для POST-форм используется CSRF-токен:

```php
csrf_field()
```

На сервере токен проверяется перед выполнением действия. Это защищает формы от отправки с внешних сайтов.

## Rate limiting и DDoS-защита

В рамках дипломного проекта реализована базовая прикладная защита от злоупотреблений:

- таблица `rate_limits`;
- функция `check_rate_limit()`;
- ограничение попыток входа;
- ограничение регистрации;
- ограничение восстановления пароля;
- ограничение повторной отправки email-кода;
- ограничение подписки на новости;
- запись блокировок в `security_events`.

Важно: полноценная DDoS-защита обычно реализуется не только в коде приложения, а на уровне инфраструктуры.

Для production рекомендуется:

- reverse proxy, например Nginx;
- лимиты `limit_req` и `limit_conn`;
- CDN/WAF, например Cloudflare или Yandex Smart Web Security;
- ограничение размера тела запроса;
- мониторинг всплесков запросов;
- блокировка IP по логам;
- fail2ban или аналогичный инструмент;
- CAPTCHA после серии неудачных попыток.
